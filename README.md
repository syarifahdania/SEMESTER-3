## 🛠️ Software Engineering · WBL · SECP3204

Software Engineering introduced the professional discipline behind building software at scale — not just writing code, but managing the full lifecycle from requirements to deployment within a team. Agile methodologies, architecture planning, testing strategies, and version control practices formed the core of the curriculum, with the WBL component grounding everything in real project constraints.

The most enduring lesson was that most software failures are communication failures, not technical ones. Misunderstood requirements, undocumented decisions, and untested assumptions cause more project damage than any bug. Writing documentation for teammates rather than just for myself, and learning to surface assumptions early rather than late, changed how I contribute to collaborative work. These habits have carried into every group project and professional engagement since.

## 🗂️ Data Structure and Algorithm · SECJ2013

If there is one course that most directly sharpened my technical thinking, it is this one. Studying arrays, linked lists, trees, heaps, hash tables, and sorting and searching algorithms — and implementing them from scratch rather than importing a library — gave me a level of intuition about computational trade-offs that no amount of high-level API usage could have produced.

The Big-O framework for analysing algorithmic efficiency became a lens I now apply instinctively. Before this course, I would write code that worked. After it, I began asking whether the solution was the right one for the scale and structure of the data involved. That distinction — between code that runs and code that scales — is one of the most important shifts in a data engineer's development, and this course is where it happened for me.

## 🧩 System Development Technology · WBL · SECP3723

This course introduced the modern frameworks and tools that bridge academic programming with professional system development — RESTful API design, web frameworks, and the integration patterns that connect front-end interfaces to back-end services and data layers.

The WBL format meant I was navigating real codebases, real documentation, and real integration errors rather than controlled exercises. That experience — deciding which tool to use with incomplete information, debugging across system boundaries, making architectural choices under time pressure — is qualitatively different from coursework, and it accelerated my professional readiness in a way that classroom-only learning could not have. I want to deepen this further with containerisation and cloud deployment, which are now standard expectations in data engineering roles.

## 🗄️ Database · WBL · SECP2523

Databases are the infrastructure on which data engineering rests, and this course built that foundation properly. Relational design, normalisation, SQL across its full range — DDL, DML, DCL — transactions, and indexing formed a comprehensive curriculum that the WBL component made immediately applied.

Normalisation clicked when I saw firsthand the data anomalies — update anomalies, insertion anomalies, deletion anomalies — that emerge from poorly designed schemas. Writing complex JOIN queries and then optimising them with execution plans and indexes revealed how much engineering sits beneath what looks like a straightforward query. I came away with SQL as a genuine professional tool rather than a subject to pass, and I now want to extend that to cloud data warehousing platforms like Snowflake and BigQuery, where the same principles apply at a different scale.

## 🤝 Caring Society · ULRF2252

This course moved me outside the technical environment I spend most of my time in and into community engagement with people whose lives are shaped — often invisibly — by the systems that professionals like me design and deploy.

The impact was more lasting than I expected. Seeing technology through the eyes of people who are not tech-savvy, and who may be underserved or excluded by poorly designed systems, reinforced a conviction I now hold professionally: accessibility and inclusivity are not optional features. They are the measure of whether a system has actually solved the problem it set out to solve.

## 🌐 Network Communications · SECR1213

This course covered the infrastructure that makes distributed computing possible — the OSI and TCP/IP models, IP addressing, subnetting, routing protocols, DNS, and the fundamentals of network security. It required careful, detail-oriented study, and it rewarded that investment with a much clearer picture of how data moves across the internet and within enterprise environments.

The relevance to data engineering became clear when I connected network concepts — latency, throughput, packet loss — to the performance characteristics of distributed data systems. A data pipeline is only as fast as the network it runs on, and understanding the underlying mechanics means I can reason about performance bottlenecks rather than simply accepting them. I want to build on this with more hands-on experience in cloud networking — VPCs, load balancers, and content delivery — which are the practical manifestations of these principles in modern infrastructure.
